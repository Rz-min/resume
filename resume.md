## MarketData サービスの Rust リプレイス

- デスクトップアプリサービス・Web サービス
- [公開情報](https://go.sbisec.co.jp/lp/lp_hyper_sbi2_211112.html)
- 期間: 2024/12~
- チーム規模: 1 人
- 担当: アーキテクチャー・設計・実装・テスト
- オンプレミス
- 利用技術:
  - 言語: Rust, C
  - その他: axum, opentelemetry, zmq, messagepack, pubsub、puppet
- 概要:

Hyper SBI2(トレーディングツール)内で主に利用されている、ランキングサービスのリプレイスプロジェクトです。
既存の C++で作成されていたサービスを Rust を使用して、マイクロサービスアーキテクチャーでのリプレイスを行いました。

<details>
<summary>詳細・工夫・ポイント</summary>

### アーキテクチャー・設計

- マイクロサービス化:  
役割やライフサイクルごとにプロセス分割を行い、各マイクロサービスでスケーラビリティ・メンテナンス性を確保できるようにしました。
- デスクトップアプリケーションの開発を行っているチームと責任分界点が曖昧になっていたので、interface を Web 技術にして、API を提供するようにして明確化しました。
- Web サーバー部分には 今後の AWS 以降に伴う技術変更などに対応しやすいようにレイヤードアーキテクチャーでの実装で技術やロジックの関心を分割しました。
- design document を書くことで、チームメンバーとの議論・レビューの効率化を図り、実装方針の合意形成を行いました。

### 実装

- opentelemetry でカスタム計装の実装と datadog 連携を導入して、Rustアプリケーションにオブザーバビリティ監視の導入を始めて行いました。
- ランキング種別の追加に伴い、指数関数的なデータ増加が問題となっていたため、データの持ち方を見直して、25%ほどデータ量を削減しました。
- c 言語で select 関数を用いて fd の監視が行われていた部分を Rust で epoll を用いた実装に変えることで io の多重化とスレッドブロックを排除しパフォーマンスの改善を行いました。
- テストツールの作成: [ratatui](https://ratatui.rs/)を利用して TUI ベースのリアルタイムテストツールの作成を行いました。
- ランキングサービスのフィルター機能が複数のチャンネルによって変わるような仕様となっていたため、動的ライブラリのPluginとしてInterfaceを提供し、各チャネルで独自に実装することができる様にして柔軟性を実現しました。

</details>

## 「金・銀・プラチナ」 システムリプレイス

- Web サービス
- 公開情報: [金・銀・プラチナ](https://site0.sbisec.co.jp/marble/gbi/top.do?)
- 期間: 2024/11~
- チーム規模: 2 人(全体 10)
- 担当: チームリード・Cloud Design・設計
- 利用技術:
  - 言語: C++, python
  - クラウド: AWS
  - その他: datadog, CDK, Quickfix
- 概要:

SBI 証券のメインのサイトで運用している、金・銀・プラチナ取引システムのリプレイスプロジェクトです。
貴金属現物取引プラットフォームへの注文取次やリアルタイム為替の計算を担うシステムのバックエンドエンジニアとして参画しました。
システム設計・cdk のコーディング・ソフトウェアレビュー・プロジェクトのチームリードを主な役割として業務を遂行しました。

<details>
<summary>詳細・工夫・アピールポイント</summary>

### システム設計・ソフトウェア設計

AWS へのリフトアンドシフトがこのプロジェクトの目的であったため、クラウドネイティブ化を含むアーキテクチャーの大きな見直しではなく、コンテナ化やデータストア変更などの小規模なアーキテクチャーの変更で運用負荷を減らすように設計を行ないました。

- 軽量な BaseImage(Distroless)でのコンテナ化

金融業界ということもあり、セキュリティ要件は厳しく、脆弱性診断ツールのアラートは迅速に対応する必要がありました。
不要なパッケージをイメージに入れることは脆弱性に触れる可能性が上がるので出来るだけ不要なパッケージを省くようにしました。
コンテナのベースイメージ管理は組織的や権限的な理由で難易度が高かったが、こちらを頑張ることで運用負荷が減るので優先した。

- Config の変更

既存で使用されていたソフトウェアの config は xml で読みににくい、かつ、メンテナンスされていない自作ライブラリの組み込みで使用されていました。
また、oss の xml パーサーは脆弱性があり、バージョンを上げる必要もありました。
メンテナンスの問題から切り離すために、自作ライブラリの脱却と yaml での config に変更を行うことで、運用負荷を減らすようにしました。

- ログの構造化

SideCar パターンを利用した datadog へのログ連携を行うために、Json でログを構造化するようにしました。
アプリケーションログ、アクセスログ、プロトコル特有のログ(Quickfix log)が、それぞれの format と logger で実装されていたので、集約と format 統一、構造化することで同じ stdout に流すように変更し、fluent-bit 側でログの解析と振り分けを行うことでシンプルな作りにしました。
また、一部パースで CPU を利用する部分があったので、datadog log pipeline によるパースに寄せることで懸念点を回避しました。

</details>

## 日米株取引アプリ統合

- スマホアプリサービス
- 公開情報: Web[サイトリニューアル](https://www.sbigroup.co.jp/news/pr/2024/0625_14740.html)
- 期間: 2023/12~2024/09
- チーム規模: 3 人(全体 20 程)
- 担当: チームリード・cloud design・設計・実装・テスト
- 利用技術:

  - 言語: C++・Go・Python・(Rust)
  - フレームワーク: drogon, gqlgen
  - クラウド: AWS
  - その他: CDK, datadog

- 概要:

日株・米株のスマホ取引アプリの統合プロジェクト、バックエンド側のゲートウェイを提供するサービスにチームリードとして参画しました。
オンプレミスからの一部マイグレーションを含むプロジェクトであり、大規模なアーキテクチャー変更の第一段階としてのプロジェクトでもありました。
また、利用者・トラフィック量とも大規模なシステムでもありました。

- 担当業務:

ゲートウェイサービスのチーム推進が主な役割で、タスクの切り出し・管理・設計・実装・他チーム連携・各種ソフトウェアのコードレビューを担当しました。
また、IaC(cdk python)・パフォーマンステスト・運用周りのコードと管理用 API などインフラや運用部分の実装タスクを実際に手を動かす部分としては担当していました。

<details>
<summary>詳細・工夫・アピールポイント</summary>

### AWS 設計

waf や API gateway などのサービスのコストが高く、ポリシーの制御やレートリミットや通信経路の制限など利点はあったが、アプリケーション側の実装に同機能を寄せることで、ネットワークコストを35%ほど試算では削減できました。
また、 cloudfront の vpc origin や ALB/NLB の暖気 API など当時欲しかった機能がリリースされてきているので、それらに置き換えたいところです。

### ハイパフォーマンスサーバー

大規模かつ不定期なスパイク型のトラフィック特性のサービスであったため、プロセス単体でのパフォーマンスが重要でした。
c++ drogon を利用することだったり、lock する部分を busy look での実装にしたりと工夫することで、数万 k rps ほどのパフォーマンスレベルを提供することができました。
また、cpu バウンド IO バウンドをスレッドプールレベルでの分割をすることで、スレッドブロックを発生させない様に工夫しました。

### 負荷テストの効率化

過去実績としては、JMeter on EC2 を用いた負荷テストの方法が社内ではあったが、このやり方では、社内ガバナンス観点でアジリティ性がなく、十分なパフォーマンステストをチームで行うことができないと判断しました。そこで、より効率的にパフォーマンステストが行えるように、AWS の DLT(Distributed Load Testing)を提案し、DLTのcdkをカスタムとメンテナンスする様にして、合意をとりパフォーマンステストの効率化を図ることができました。
</details>

## SOR 注文ソフトウェア開発

- リアルタイム取引サービス
- 公開情報:
  [SBI クロス](https://www.sbigroup.co.jp/news/pr/2024/0419_14589.html)
- 期間: 2024/02~2024/04
- チーム規模: 3 人 (全体 10 人程)
- 担当: 設計・実装・テスト
- 利用技術: Rust・C++
- 概要:

ダークプール取引(立会外市場)の追加に伴い SOR (Smart Order Routing)システムの新規機能追加と一部リプレイスのプロジェクトを行いました。
私は、市場データ周りを担当し、パフォーマンス懸念のあった板情報受信モジュールの Rust でのリプレイスを担当しました。

<details>
<summary>詳細・工夫・アピールポイント</summary>

### 貢献

- system v IPC の共有メモリとカウンティングセマフォを利用した実装に変更することで、リアルタイム更新のパフォーマンス改善をしました。
- 冗長なバイナリサーチからメモリオフセットの位置をインメモリにキャッシュする機構を入れることで計算量を減らした。
- 状態管理機構をビッドフラグによる実装に変更することで、ロジックを集約して、簡潔に実装しました。
- 全体では、60k tps → 150k tps へのパフォーマンス改善が記録できました。

</details>
