- オンプレミス
- 利用技術:

  - その他: axum, opentelemetry, zmq, messagepack, pubsub、puppet
- 概要:

Hyper SBI2(トレーディングツール)内で主に利用されている、ランキングサービスのリプレイスプロジェクト。
既存の C++で作成されていたサービスを Rust を使ってマークローサービスアーキテクチャーでのリプレイスを行った。
設計・実装・テストを一貫して担当した。

<details>
<summary>詳細・工夫・ポイント</summary>

### アーキテクチャー・設計

- マイクロサービス化: 役割や機能ごとにプロセス分割を行い、各サービスでスケーラビリティ・メンテナンス性を高くした。
- デスクトップアプリケーションの開発を行っている部署と責任分界点が曖昧になっていたので、interface を web 技術にして、API を提供するようにして明確化した。
- web サーバー部分の実装には 今後の AWS 以降に伴う技術変更などに対応しやすいようにレイヤードアーキテクチャーでの実装を行った。
- design document を書いて、チームメンバーと議論する様にした。

### 実装

- opentelemetry でカスタム計装の実装と datadog 連携を導入
- データの持ち方を見直して、指数関数的に増えていたデータ量を抑える様に変更した。
- c 言語で select 関数を用いて fd の監視が行われていた部分を Rust で epoll を用いた実装に変えることで io の多重化とスレッドブロックを排除しパフォーマンスの改善を行った。
- テストツールの作成: [ratatui](https://ratatui.rs/)を利用して TUI ベースのリアルタイムテストツールの作成

- 期間: 2024/11~
- チーム規模: 2 人(全体 10)
- 担当: チームリード・Cloud Design・設計
- 利用技術:
  - 言語: C++, python
  - クラウド: AWS
  - その他: datadog, CDK, quickfix
- 概要:

SBI 証券のメインのサイトで運用している、金・銀・プラチナ取引システムのリプレイスプロジェクト。
貴金属現物取引プラットフォームへの注文取次やリアルタイム為替の計算を担うシステムのバックエンドエンジニアとして参画。
システム設計・cdk のコーディング・ソフトウェアレビュー・プロジェクトのチームリードを主な役割として業務を遂行した。

<details>
<summary>詳細・工夫・アピールポイント</summary>

### システム設計・ソフトウェア設計

AWS へのリフトアンドシフトがこのプロジェクトの目的であったため、クラウドネイティブ化を含むアーキテクチャーの大きな見直しではなく、コンテナ化やデータストアの変更などの小規模なアーキテクチャーの変更で運用負荷を減らすように設計を行なった。

- 軽量な BaseImage(Distroless)でのコンテナ化

金融業界ということもあり、セキュリティ要件は厳しく、脆弱性診断ツールのアラートは迅速に対応する必要があった。
不要なパッケージをイメージに入れることは脆弱性に触れる可能性が上がるので出来るだけ不要なパッケージを省くようにした。
コンテナのベースイメージ管理は組織的や権限的な理由で難易度が高かったが、こちらを頑張ることで運用負荷が減るので優先した。

- Config の変更

既存で使用されていたソフトウェアの config は xml で読みににくい、かつ、メンテナンスされていない自作ライブラリの組み込みで使用されていた。
また、c++の oss の xml パーサーは脆弱性があり、バージョンを上げる必要もあった。
メンテナンスの問題から切り離すために、自作ライブラリの脱却と yaml での config に変更を行った。

- ログの構造化

SideCar パターンを利用した datadog へのログ連携を行うために、Json でログを構造化するようにした。
アプリケーションログ、アクセスログ、プロトコル特有のログ(quickfix log)が、それぞれの format と logger で実装されていたので、集約と format 統一、構造化することで同じ stdout に流すようにした。
fluent-bit 側でログの解析と振り分けを行うことでシンプルな作りに変更した。また、一部パースで CPU を利用する部分があったので、datadog log pipeline によるパースに寄せることで懸念点を回避した。

</details>

## 日米株取引アプリ統合

- スマホアプリサービス
- 公開情報: web[サイトリニューアル](https://www.sbigroup.co.jp/news/pr/2024/0625_14740.html)
- 期間: 2023/12~2024/09
- チーム規模: 3 人(全体 20 程)
- 担当: チームリード・cloud design・設計・実装・テスト
- 利用技術:

  - 言語: C++・Go・Python・(Rust)
  - フレームワーク: drogon, gqlgen
  - クラウド: AWS
  - その他: CDK, datadog

- 概要:

日株・米株のスマホ取引アプリの統合プロジェクト、バックエンド側のゲートウェイを提供するサービスにチームリードとして参画した。
オンプレミスからの一部マイグレーションを含むプロジェクトであり、大規模なアーキテクチャー変更の第一段階としてのプロジェクトでもあった。
また、利用者・トラフィック量とも大規模なシステムである。

- 担当業務:

ゲートウェイサービスのチーム推進が主な役割で、タスクの切り出し・管理・設計・実装・他チーム連携・各種ソフトウェアのコードレビューを担当した。
また、IaC(cdk python)・パフォーマンステスト・運用周りのコードと管理用 API などインフラや運用部分の実装タスクを実際に手を動かす部分としては担当した。

<details>
<summary>詳細・工夫・アピールポイント</summary>

### AWS 設計

waf や API gateway などのネットワークサービスのコストが高く、ポリシーの制御やレートリミットなど利点はあったが、アプリケーション側に寄せることでコストとのバランスをとって設計した。
結果として、試算では、35%ほどコストを下げることができた。
また、 cloudfront の vpc origin や ALB/NLB の暖気 API など当時欲しかった機能がリリースされてきているので、それらに置き換えたいところである。

### ハイパフォーマンスサーバー

大規模かつ不定期なスパイクリクエストのフィック特性のサービスであったため、プロセス単体でのパフォーマンスが必要であった。
c++ drogon を利用することだったり、lock する部分を busy look での実装にしたりと工夫することで、数万 k rps ほどのパフォーマンスレベルを提供することができた。
また、cpu バウンド IO バウンドをスレッドプールレベルでの分割をすることで、スレッドブロックを発生させない様に工夫した。

### 負荷テストの効率化

実績としては、JMeter on EC2 を用いた負荷テストの方法が社内ではあったが、このやり方では、社内ガバナンス観点でアジリティがなく、十分なパフォーマンステストをチームで行いたかったので、より効率的にパフォーマンステストが行えるように、AWS の DLT(Distributed Load Testing)を提案して実現した。

</details>
